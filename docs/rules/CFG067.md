# CFG067 — hooks defined in a project-scoped settings file

**Severity:** `warn`
**OWASP:** [LLM03:2025 – Supply Chain](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM03_2025-Supply_Chain.html)
**CVE:** CVE-2025-59536

## What cfgaudit checks

Claude Code hooks run automatically on lifecycle events. In a user's global `~/.claude/settings.json` they were placed there by the user themselves. In a **project**-scoped file (`.claude/settings.json` or `.claude/settings.local.json`) they were placed there by whoever committed to the repository — and they execute on any developer who opens the project, including ones who have never used Claude Code and don't know hooks exist (CVE-2025-59536).

cfgaudit flags the **presence** of any hooks in a project-scoped settings file, regardless of their content. This is a separate, lower-severity signal from the content checks: CFG008 / CFG014 / CFG015 / CFG037 / CFG038 / CFG039 still independently flag dangerous hook commands. The rule does **not** fire on the user-global `~/.claude/settings.json` (hooks there are intentional) or on plugin `hooks.json` files.

## Trigger example

```json
{
  "hooks": {
    "PreToolUse": [
      { "matcher": "*", "hooks": [{ "type": "command", "command": "./scripts/setup.sh" }] }
    ]
  }
}
```

Committed as `.claude/settings.json` → `warn`.

## Safe alternative

Keep hook definitions out of committed project config. Put them in your user-global `~/.claude/settings.json`, or document them as a reviewed onboarding step so each developer opts in knowingly.
