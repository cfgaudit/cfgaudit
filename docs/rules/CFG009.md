# CFG009 — hook command interpolates a shell variable

**Severity:** `warn`
**OWASP:** [LLM01:2025 – Prompt Injection](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM01_2025-Prompt_Injection.html)

## What cfgaudit checks

cfgaudit scans every command string under `hooks.*[].hooks[].command` in `settings.json` and flags any reference to a shell variable: `$NAME` or `${NAME}`.

Hook commands run with the privileges of the Claude Code process and fire automatically on lifecycle events. Whenever part of a hook command is interpolated from a variable, that value may originate from agent output, tool output, or filenames the agent chose — i.e. attacker-influenced content. Crafted input can break out of unquoted arguments or escape quotes when shells re-parse strings, turning the hook into an arbitrary-command-execution sink.

Command substitution (`$(...)`) and positional parameters (`$1`, `$@`, `$?`) are intentionally **not** matched by this rule — they are tracked separately (issue #38).

## Trigger example

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          { "type": "command", "command": "notify-send \"$TOOL_OUTPUT\"" }
        ]
      }
    ]
  }
}
```

## Safe alternative

Prefer fixed commands. When the hook genuinely needs context, pass it via stdin rather than as a shell-expanded argument:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          { "type": "command", "command": "/usr/local/bin/notify-from-stdin" }
        ]
      }
    ]
  }
}
```

If you must interpolate, ensure the value is single-quoted on the shell side and the receiving program treats the argument as untrusted data, not as a shell fragment.
