# CFG061 — Gemini CLI sandbox weakened (`tools.sandbox*`)

**Severity:** `error` (broad `sandboxAllowedPaths`) · `warn` (`sandboxNetworkAccess`)
**OWASP:** [LLM06:2025 – Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

Gemini CLI runs tools in a sandbox; two `settings.json` fields can weaken it (the Gemini analog of CFG022):

| Field | Risk | Severity |
|-------|------|----------|
| `tools.sandboxAllowedPaths` containing `/`, `~`, `$HOME`, `..` (or `/*`, `~/*`, `../…`) | exposes the filesystem root / home directory to sandboxed tools — path isolation is gone | `error` |
| `tools.sandboxNetworkAccess: true` | sandboxed tools get unrestricted network egress — an exfiltration / SSRF channel | `warn` |

cfgaudit checks `.gemini/settings.json` (project) and `~/.gemini/settings.json` (with `--user`).

## Trigger example

```json
{ "tools": { "sandboxAllowedPaths": ["/"], "sandboxNetworkAccess": true } }
```

→ `error` (root path) + `warn` (network access).

## Safe alternative

Scope the allow-list to the directories the tools actually need, and leave the sandbox network-isolated:

```json
{ "tools": { "sandboxAllowedPaths": ["./build", "./.cache"], "sandboxNetworkAccess": false } }
```
