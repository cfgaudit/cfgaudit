# CFG075 тАФ MCP server disables TLS certificate verification (MITM kill switch)

**Severity:** `error`
**OWASP:** [LLM02:2025 тАУ Sensitive Information Disclosure](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM02_2025-Sensitive_Information_Disclosure.html)
**OWASP MCP:** [MCP01:2025 тАУ Token Mismanagement & Secret Exposure](https://owasp.org/www-project-mcp-top-10/) тАФ provisional (MCP Top 10 v0.1)

## What cfgaudit checks

An MCP server launched with TLS certificate verification turned off silently accepts MITM'd TLS connections to its upstream. The endpoint can be `https://`, but if the process trusts any certificate, an on-path attacker reads and rewrites the traffic тАФ including the agent's context and any forwarded credentials. This defeats the remote-URL / cleartext hardening cfgaudit already enforces (CFG049 remote URL, CFG021 proxy, CFG068 templated cred to cleartext, CFG069 log redaction).

cfgaudit flags an MCP server (in `settings.json` `mcpServers`, `.mcp.json`, or any cross-agent equivalent) whose `env` or `args` disables certificate verification. These are deterministic literal key/value (or arg) patterns in a committable config.

### `env` kill switches

| Key | Disabling value |
|-----|-----------------|
| `NODE_TLS_REJECT_UNAUTHORIZED` | `0` |
| `PYTHONHTTPSVERIFY` | `0` |
| `GIT_SSL_NO_VERIFY` | `1`/`true` |
| `SSL_VERIFY`, `VERIFY_SSL`, `*_SSL_VERIFY`, `*_TLS_VERIFY` | `false`/`0` |
| `NPM_CONFIG_STRICT_SSL` | `false` |
| `PGSSLMODE` | `disable` |
| `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE` | empty / `/dev/null` |
| any value containing `sslmode=disable` (DB connection string) | тАФ |

### `args` kill switches

| Token | Tool |
|-------|------|
| `--insecure` | curl / httpie / generic |
| `--no-check-certificate` | wget |
| `-k` | only when the server `command` is `curl`/`wget` (ambiguous otherwise) |

A templated / `${VAR}` / `$(тАж)` value is **not** flagged тАФ it is resolved at runtime, consistent with CFG050/054. Disjoint from CFG020 (`NODE_OPTIONS`/`LD_PRELOAD` code injection) and CFG021 (proxy routing) тАФ neither touches cert validation, so there is no double report.

## Trigger example

```json
{
  "mcpServers": {
    "api": {
      "command": "node",
      "args": ["server.js"],
      "env": { "NODE_TLS_REJECT_UNAUTHORIZED": "0" }
    }
  }
}
```

тЖТ `error`: TLS verification is off, so an `https://` upstream is MITM-able regardless of scheme.

## Safe alternative

Leave certificate verification on. If the upstream uses a private CA, point the client at that CA bundle (e.g. `NODE_EXTRA_CA_CERTS=/path/to/ca.pem`, `REQUESTS_CA_BUNDLE=/path/to/ca.pem`) instead of disabling verification.

```json
{ "mcpServers": { "api": { "command": "node", "args": ["server.js"],
  "env": { "NODE_EXTRA_CA_CERTS": "/etc/ssl/corp-ca.pem" } } } }
```
