# CFG064 ‚Äî Codex CLI sandbox disabled (`sandbox_mode: danger-full-access`)

**Severity:** `error`
**OWASP:** [LLM06:2025 ‚Äì Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

OpenAI Codex CLI's `sandbox_mode` (in `.codex/config.toml`) controls how tool execution is confined. The value **`danger-full-access`** disables the sandbox entirely ‚Äî tools run with full filesystem and network access ‚Äî the analog of weakening Claude Code's sandbox (CFG022). Combined with `approval_policy = "never"` (CFG063) it is a fully unattended, unsandboxed agent.

The safe values are `read-only` and `workspace-write`.
## Where cfgaudit looks

Codex config is **project-merged**, not user-global. Besides the user file it loads a repo-local `.codex/config.toml` ‚Äî upstream resolves it both at the git root and by walking parent directories (`codex-rs/config/src/loader/mod.rs`, layer order `cwd` ‚Üí `tree` ‚Üí `repo`).

cfgaudit therefore scans:

| Path | Scope | Flag |
|---|---|---|
| `<repo>/.codex/config.toml` | project | always |
| `~/.codex/config.toml` (`$CODEX_HOME`) | user | `--user` |

**A committed `.codex/config.toml` is the case this rule exists for**, and it is the one cfgaudit missed until #388 ‚Äî the rule was correct but only ever looked at the user file.

The bare `${PWD}/config.toml` layer is deliberately not scanned: the filename is too generic (Hugo, Zola and others use it), so decoding every repo's `config.toml` as Codex config would turn an unrelated file into a scan error for no coverage gain.

### Keys Codex refuses from a project layer

Upstream guards a subset against repo contents (`PROJECT_LOCAL_CONFIG_DENYLIST`), commented *"Project-local config comes from repository contents, so it should not get to choose where a user's credentials are sent or which local commands are run."* It covers `openai_base_url`, `chatgpt_base_url`, `apps_mcp_product_sku`, `model_provider(s)`, `notify`, `profile(s)`, the two experimental realtime URLs, and `otel`.

cfgaudit drops those from a project file so it never reports configuration the CLI ignores ‚Äî this is why a committed `notify` is not treated as a command site and a committed `chatgpt_base_url` does not trigger CFG071.

`approval_policy`, `sandbox_mode` and `mcp_servers` are **not** on that denylist, which is exactly why they are worth flagging in a committed file.

### Trust gate

Project layers are *"loaded but disabled when the directory is untrusted"*. That is a real mitigation, and the same one CFG086 records for Cursor's workspace trust ‚Äî and the same reasoning applies: trust is granted once, often reflexively, and thereafter every clone is covered. It caps the blast radius; it is not a reason to leave the setting in the repository.

## Trigger example

```toml
sandbox_mode = "danger-full-access"
```

‚Üí `error`.

## Safe alternative

```toml
sandbox_mode = "workspace-write"
```
