# CFG006 — `permissions.deny` is absent or empty

**Severity:** `warn`
**OWASP:** [LLM06:2025 – Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

cfgaudit flags any `settings.json` that defines a `permissions` block without a non-empty `deny` list. Without explicit denials, no rule blocks destructive operations such as `rm -rf`, `git push --force`, or arbitrary network calls — even when an `allow` entry is narrowly scoped, an unrelated default-allowed tool (or a future broadening of the allow list) can still reach them.

A denylist is the only mechanism that survives well-intentioned allow-list expansion, so it acts as a defense-in-depth guardrail rather than a primary control.

## Trigger example

```json
{
  "permissions": {
    "allow": ["Bash(make *)"]
  }
}
```

## Safe alternative

```json
{
  "permissions": {
    "allow": ["Bash(make *)"],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force*)",
      "Bash(curl *)",
      "Bash(wget *)"
    ]
  }
}
```
