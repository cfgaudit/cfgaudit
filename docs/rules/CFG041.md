# CFG041 — `permissions.deny` does not restrict access to `.env` files

**Severity:** `error`
**OWASP:** [LLM02:2025 – Sensitive Information Disclosure](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM02_2025-Sensitive_Information_Disclosure.html)

## What cfgaudit checks

`.env` files routinely hold database passwords, API keys, and other credentials. `permissions.deny` is Claude Code's documented, harness-enforced mechanism for blocking file access. cfgaudit flags a `permissions.deny` block that **exists** but contains **no** pattern covering `.env` files.

A pattern counts as sufficient coverage if it targets `.env`:

```
.env
.env.*
*.env
**/.env
**/.env.*
```

A completely **absent** `permissions.deny` block is reported by CFG006, not here.

A **deny-all** entry also counts as coverage and suppresses this finding: `Read(*)` / `Read(**)` / `Read(**/*)` (blocks every read, any version), or the bare `"*"` deny-all-tools glob — but `"*"` only on Claude Code **≥ 2.1.166**, where that glob was introduced. On older releases `"*"` is an unknown tool name and a no-op, so the finding still fires (add an explicit `.env` pattern or upgrade). When the Claude Code version is unknown, `"*"` is treated as covering.

## Trigger example

```json
{
  "permissions": {
    "deny": ["Bash(rm -rf *)"]
  }
}
```

→ `error` (deny exists but no `.env` coverage).

## Safe alternative

```json
{
  "permissions": {
    "deny": ["Read(**/.env)", "Read(**/.env.*)"]
  }
}
```
