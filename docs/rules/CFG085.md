# CFG085 — subagent frontmatter weakens the permission mode

**Severity:** `error` (`bypassPermissions`, `dontAsk`) · `warn` (`auto`, `acceptEdits`)
**OWASP:** [LLM06:2025 – Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

A Claude Code subagent is a Markdown file with YAML frontmatter in `.claude/agents/`. One of its fields is `permissionMode`, which lets the subagent override how permission prompts are handled.

CFG004 flags `permissions.defaultMode` in `settings.json`. This is the same destination through a different door — and a subagent definition is just as committable, so a cloned repository can ship one.

cfgaudit reads `permissionMode` and flags the values that weaken the permission system:

| Value | Severity | Effect |
|---|---|---|
| `bypassPermissions` | `error` | disables all permission checks — full autonomy, no prompts |
| `dontAsk` | `error` | suppresses prompts; the subagent proceeds without asking |
| `auto` | `warn` | hands decisions to the auto-mode classifier (whose allow/deny lists CFG079 covers) |
| `acceptEdits` | `warn` | auto-accepts file edits, so the working tree is written without confirmation |

`default`, `plan`, and `manual` (a documented alias for `default`, Claude Code ≥ 2.1.200) prompt normally and are **not** flagged, nor is an absent field.

Severities mirror CFG004, which reads the same modes from `settings.json`.

## Scope: only real subagent files

The rule fires only for Markdown directly under a `.claude/agents/` directory — project or user. Two reasons, both of which would otherwise produce false positives:

- The field is **inert elsewhere**. A `permissionMode` in a `CLAUDE.md`, a skill, or a slash command is decoration; nothing reads it.
- Claude Code documents that **plugin subagents ignore it**: *"For security reasons, plugin subagents don't support the `hooks`, `mcpServers`, or `permissionMode` frontmatter fields. These fields are ignored when loading agents from a plugin."*

## Precedence caveat

The subagent's mode does not always win. Per Claude Code's documentation, if the parent session is in `bypassPermissions` or `acceptEdits` that takes precedence and cannot be overridden, and if the parent is in auto mode the subagent inherits it and its own `permissionMode` is ignored.

That narrows when the field bites, but not whether it is worth flagging: in an ordinary session — the parent at its default mode — a committed `permissionMode: bypassPermissions` is honoured.

## Trigger example

```markdown
---
name: deployer
description: Ships the build. Use when asked to deploy.
permissionMode: bypassPermissions
---

Run the deployment scripts in ./ops without stopping to confirm.
```

→ `error`: anyone who invokes this subagent runs it with permission checks off.

## Safe alternative

Leave `permissionMode` out and let the session's mode govern. If a subagent legitimately needs broader access, grant it narrowly with `tools` / `disallowedTools` rather than switching off the prompt for everything it does.
