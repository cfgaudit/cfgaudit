# CFG063 тАФ Codex CLI auto-approves commands (`approval_policy`)

**Severity:** `error` (`never`) ┬╖ `warn` (`on-failure`)
**OWASP:** [LLM06:2025 тАУ Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

OpenAI Codex CLI's `approval_policy` (in `.codex/config.toml`) controls when the agent asks before running commands тАФ the analog of Claude Code's `defaultMode`. Two values remove the human-in-the-loop:

| Value | Meaning | Severity |
|-------|---------|----------|
| `never` | Never ask; **all** commands are auto-approved (non-interactive) тАФ the Codex equivalent of `defaultMode: bypassPermissions` | `error` |
| `on-failure` | DEPRECATED тАФ all commands are auto-approved and only escalated to the user **on failure** | `warn` |

The safe values are `untrusted` (only known-safe read-only commands auto-approved) and `on-request`.
## Where cfgaudit looks

Codex config is **project-merged**, not user-global. Besides the user file it loads a repo-local `.codex/config.toml` тАФ upstream resolves it both at the git root and by walking parent directories (`codex-rs/config/src/loader/mod.rs`, layer order `cwd` тЖТ `tree` тЖТ `repo`).

cfgaudit therefore scans:

| Path | Scope | Flag |
|---|---|---|
| `<repo>/.codex/config.toml` | project | always |
| `~/.codex/config.toml` (`$CODEX_HOME`) | user | `--user` |

**A committed `.codex/config.toml` is the case this rule exists for**, and it is the one cfgaudit missed until #388 тАФ the rule was correct but only ever looked at the user file.

The bare `${PWD}/config.toml` layer is deliberately not scanned: the filename is too generic (Hugo, Zola and others use it), so decoding every repo's `config.toml` as Codex config would turn an unrelated file into a scan error for no coverage gain.

### Keys Codex refuses from a project layer

Upstream guards a subset against repo contents (`PROJECT_LOCAL_CONFIG_DENYLIST`), commented *"Project-local config comes from repository contents, so it should not get to choose where a user's credentials are sent or which local commands are run."* It covers `openai_base_url`, `chatgpt_base_url`, `apps_mcp_product_sku`, `model_provider(s)`, `notify`, `profile(s)`, the two experimental realtime URLs, and `otel`.

cfgaudit drops those from a project file so it never reports configuration the CLI ignores тАФ this is why a committed `notify` is not treated as a command site and a committed `chatgpt_base_url` does not trigger CFG071.

`approval_policy`, `sandbox_mode` and `mcp_servers` are **not** on that denylist, which is exactly why they are worth flagging in a committed file.

### Trust gate

Project layers are *"loaded but disabled when the directory is untrusted"*. That is a real mitigation, and the same one CFG086 records for Cursor's workspace trust тАФ and the same reasoning applies: trust is granted once, often reflexively, and thereafter every clone is covered. It caps the blast radius; it is not a reason to leave the setting in the repository.

## Trigger example

```toml
approval_policy = "never"
```

тЖТ `error`.

## Safe alternative

```toml
approval_policy = "on-request"
```

Keep approval interactive; reserve `never` for sandboxed, non-interactive CI runs where you also constrain `sandbox_mode` (see CFG064).
