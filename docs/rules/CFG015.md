# CFG015 — hook command contains shell substitution

**Severity:** `warn` (escalates to `error` if a network call appears inside the substitution)
**OWASP:** [LLM01:2025 – Prompt Injection](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM01_2025-Prompt_Injection.html)

## What cfgaudit checks

cfgaudit scans every command-bearing key in `settings.json` — hooks (`hooks.*[].hooks[].command`) **and** the credential/runtime helpers Claude Code also executes (`apiKeyHelper`, `awsCredentialExport`, `awsAuthRefresh`, `gcpAuthRefresh`, `otelHeadersHelper`, `statusLine.command`, `fileSuggestion.command`) — for shell-evaluated substitutions:

- `$( … )` — POSIX command substitution
- `` ` … ` `` — backtick command substitution

The output of a substituted command is spliced into the shell line at runtime. When any input is attacker-controlled (filename Claude chose, tool output, an injected commit message, …), the substitution becomes a command-injection sink.

A finding is emitted at `warn` when a substitution is present and at `error` when the substituted command itself reaches the network (`curl`, `wget`, `nc`, `ncat`, `ssh`, `scp`, `rsync`, `ftp`, `telnet`, `nslookup`, `dig`, `host`). Network calls inside a substitution mean the value spliced into the hook is whatever some external endpoint chose to return.

This rule is **complementary** to:

- **CFG009** — flags plain `$VAR` / `${VAR}` interpolation (variable, not substitution).
- **CFG014** — flags `curl … | bash`-style pipelines (substitution piping into an interpreter is the same family, but the surface is different).

Multiple substitutions in one command are consolidated into a single finding; the message lists each substitution string.

## Trigger examples

```json
{
  "hooks": {
    "PostToolUse": [
      { "type": "command", "command": "notify-send \"$(cat /etc/passwd)\"" },
      { "type": "command", "command": "echo `date`" }
    ]
  }
}
```

Both fire. The first is `warn` (no network call); the second is also `warn`. Replace `cat /etc/passwd` with `curl …` and the severity becomes `error`.

## Safe alternative

Hooks should be deterministic and not depend on values evaluated at hook time. If you need dynamic data, pass it via stdin or environment variables that you set yourself (do not let attacker-controlled inputs flow into a `$(…)`):

```json
{
  "type": "command",
  "command": "/usr/local/bin/notify-from-stdin"
}
```
