# CFG048 ÔÇö `.vscode/settings.json` blanket-auto-approves agent tools

**Severity:** `error` (a workspace-applied key weakened) ┬À `warn` (protection dropped, or an upstream-inert key)
**OWASP:** [LLM06:2025 ÔÇô Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

`.vscode/settings.json` is committed into repositories and read by VS Code / Copilot **and its forks (Cursor, Windsurf)**. Several settings in the auto-approve family decide how much the agent may do unattended ÔÇö but they do **not** all take effect from a committed workspace file, and that distinction drives this rule.

| Setting | Registered scope | Applied from a committed workspace file? | Severity |
|---|---|---|---|
| `chat.tools.edits.autoApprove` | *(none)* ÔåÆ `WINDOW` | **Yes** | `error` / `warn` |
| `chat.tools.urls.autoApprove` | *(none)* ÔåÆ `WINDOW` | **Yes** | `error` |
| `chat.tools.global.autoApprove` | `APPLICATION_MACHINE` | **No** (upstream) | `warn` |
| `chat.tools.autoApprove` (legacy) | ÔÇö | ÔÇö | `warn` |

### `chat.tools.edits.autoApprove` ÔÇö the sharp one

A map of glob ÔåÆ bool deciding which file edits the agent makes without asking. VS Code's default approves everything **except** a denylist of files with immediate side effects ÔÇö `**/.vscode/*.json`, `**/.git/**`, `.env`, lockfiles, project and build files.

cfgaudit reports `error` when a committed map sets one of those protected patterns to `true`. Re-enabling `**/.vscode/*.json` in particular chains directly into CFG047: the agent can then rewrite the tasks file that runs on folder open, which is a path to unprompted code execution.

It reports `warn` when the map approves the whole tree (`**/*`) **without** restating the default denials. Whether that is exploitable depends on whether VS Code merges the committed map into its defaults or replaces them outright; the warning says so rather than asserting the worse case.

### `chat.tools.urls.autoApprove`

A map of URL pattern ÔåÆ approval controlling which endpoints chat tools may fetch unattended. cfgaudit flags only patterns with **no meaningful host restriction** (`*`, `**`, `https://*`, ÔÇĞ), which is the fetch-tool analogue of CFG040's unrestricted `WebFetch`. Committing a specific host ÔÇö an internal docs site, say ÔÇö is ordinary team configuration and is not flagged. Values may be a bool or an object of `approveRequest` / `approveResponse`; approving either direction counts.

### The blanket booleans

`chat.tools.global.autoApprove` and the older `chat.tools.autoApprove` switch off confirmation for every tool call, and per Microsoft's documentation also disable the terminal allow/deny-list.

They are reported at **`warn`, not `error`**, because upstream VS Code registers the current key with `ConfigurationScope.APPLICATION_MACHINE` ÔÇö it is *ignored* when it appears in a committed workspace file. Coverage is kept rather than dropped because forks read the same file and may honour it at workspace scope; an unverified fork behaviour is a reason to lower the severity, not to stop looking.

`.vscode/settings.json` is JSONC (comments and trailing commas), which cfgaudit tolerates; a file that still fails to parse is reported as a tool error rather than silently trusted.

## Trigger example

```jsonc
{
  // dev convenience
  "chat.tools.global.autoApprove": true
}
```

ÔåÆ `error` ÔÇö every agent tool call is auto-approved for anyone who opens the repo.

## Safe alternative

Remove the blanket setting and approve tools interactively, or scope auto-approval to specific safe commands with the terminal allow-list:

```jsonc
{
  "chat.tools.terminal.autoApprove": {
    "npm run test": true,
    "/^git (status|diff|log)/": true
  }
}
```
