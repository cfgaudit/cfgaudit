# CFG001 — `permissions.allow` contains unrestricted Bash pattern

**Severity:** `error`  
**OWASP:** [LLM06:2025 – Excessive Agency](https://owasp.org/www-project-top-10-for-large-language-model-applications/2025/LLM06_2025-Excessive_Agency.html)

## What cfgaudit checks

cfgaudit flags `permissions.allow` entries that match `Bash(*)` or `Bash(**)` — patterns that grant the agent permission to execute any shell command without a confirmation prompt. This maximises blast radius if the agent is manipulated via prompt injection or a malicious tool response.

## Trigger example

```json
{
  "permissions": {
    "allow": ["Bash(*)"]
  }
}
```

## Safe alternative

Scope permissions to the specific commands the agent needs:

```json
{
  "permissions": {
    "allow": ["Bash(go test ./...)", "Bash(make build)", "Bash(make lint)"]
  }
}
```

## Relationship to the SchemaStore schema

The [SchemaStore Claude Code settings schema](https://json.schemastore.org/claude-code-settings.json) already rejects `Bash(*)` — its `permissionRule` pattern requires the args inside parentheses to contain at least one character that is not `)`, `*`, or `?`. That check only fires in editors and tools that load the schema (VS Code IntelliSense, JSON validators in CI plugins, etc.) and only if the developer has not silenced the warning.

CFG001 catches the same misconfiguration in every other context: file-system audits, pre-commit hooks, scans of checked-out repos by people who do not use the schema-aware editor, and any future relaxation of the upstream schema. Treat the two as complementary, not redundant.
